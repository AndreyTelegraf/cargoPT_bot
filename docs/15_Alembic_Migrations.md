# CargoPT Bot — Alembic Migrations v1

## Purpose

This document defines migration strategy for CargoPT Bot.

The goal is:

- predictable schema evolution
- safe deployment
- reversible releases
- compatibility between environments
- reliable production upgrades

## Technology stack

Recommended stack:

- SQLAlchemy 2.x
- Alembic
- PostgreSQL
- SQLite for local development only

Alembic is the single source of truth for schema evolution.

Manual schema changes are prohibited.

## Migration principles

Every schema change must be implemented through migration.

Never:

- modify production schema manually
- modify tables without migration
- edit previously executed migrations

Once a migration reaches production:

- treat it as immutable

## Repository structure

Recommended structure:

migrations/
  env.py
  script.py.mako
  versions/

Every migration file lives inside:

migrations/versions/

## Naming convention

Migration filename:

YYYYMMDD_HHMM_short_description.py

Examples:

20260622_1200_initial_schema.py
20260624_1830_add_request_photos.py
20260630_0915_add_schedule_layer.py

Migration IDs must remain unique.

## Migration ownership

Migrations are generated from ORM models.

Flow:

ORM models
    ->
Alembic migration
    ->
Database schema

Database schema must never become the source of truth.

## Foreign key policy

Foreign keys should be explicit.

Do not rely only on application-level references.

MVP foreign key behavior:

- avoid cascade delete for business records
- prefer restricted deletes
- use nullable references where operationally optional

Examples:

- request_assignment.vehicle_id may be nullable
- request_assignment.driver_id may be nullable
- request_offer.vehicle_id may be nullable

## Nullable policy

Nullable fields are allowed when:

- data may not be known at request creation
- field is optional in MVP
- field belongs to future schedule/calendar support
- field is a parsed derivative of raw user input

Non-nullable fields should include:

- primary identifiers
- required foreign keys
- status fields
- created_at
- core ownership fields

## Autogenerate policy

Alembic autogenerate is allowed, but never trusted blindly.

Autogenerated migration must be reviewed before commit.

Review checklist:

- table names are correct
- column names are correct
- nullable flags are correct
- foreign keys are present
- indexes are present
- unique constraints are present
- downgrade exists
- no accidental table drop
- no accidental column drop
- no unintended type change

Autogenerate command:

alembic revision --autogenerate -m "message"

Then inspect generated file manually.

## Manual migration edits

Manual edits are allowed when autogenerate is incomplete.

Common manual edits:

- explicit index names
- explicit unique constraint names
- data backfill
- enum/check constraint improvements
- SQLite compatibility adjustments

Manual edits must be simple and auditable.

## Upgrade policy

Every migration must define upgrade().

upgrade() must:

- create or alter schema deliberately
- preserve existing data where possible
- avoid long-running operations in production
- be safe to run once

upgrade() must not:

- silently delete business data
- assume empty production database unless initial migration
- perform unrelated business changes

## Downgrade policy

Every migration should define downgrade() for MVP.

downgrade() must:

- reverse schema change where reasonable
- drop dependent objects in safe order
- not pretend to restore deleted data

If downgrade cannot safely restore data, document it clearly in the migration comment.

## Data migration policy

Some migrations may need data updates.

Examples:

- populate default status
- backfill offer_round
- normalize old values
- create derived rows

Rules:

- keep data migrations small
- make them idempotent where possible
- test on DB copy before production
- never mix unrelated schema and data changes if avoidable

## SQLite compatibility

SQLite is allowed for local development.

But PostgreSQL-readiness remains the design target.

SQLite limitations to remember:

- weaker ALTER TABLE support
- partial indexes differ
- boolean is stored differently
- foreign key enforcement requires configuration
- some constraint behaviors differ

Migrations should be tested on target database before production.

## PostgreSQL readiness

Before production launch, validate migrations against PostgreSQL.

Minimum checks:

- alembic upgrade head
- alembic downgrade base if safe
- alembic upgrade head again
- inspect constraints
- inspect indexes
- run smoke inserts

## Production safety

Before running migration on production:

1. Confirm current git commit.
2. Confirm current database URL.
3. Create database backup.
4. Confirm backup exists.
5. Run migration on staging or local copy.
6. Run smoke tests.
7. Apply migration.
8. Restart service if needed.
9. Run production smoke.

Never run production migration without backup.

## Backup policy

Before every production migration:

- create database backup
- store backup outside working tree
- include timestamp and git commit in backup name

Example backup name:

cargopt_prod_20260622_1530_before_bdcd631.sql

SQLite backup example:

cp data/cargopt.db backups/cargopt_YYYYMMDD_HHMM_before_COMMIT.db

PostgreSQL backup example:

pg_dump "$DATABASE_URL" > backups/cargopt_YYYYMMDD_HHMM_before_COMMIT.sql

## Migration deployment flow

Recommended flow:

1. git pull
2. install dependencies
3. create backup
4. run alembic current
5. run alembic upgrade head
6. run application smoke
7. restart bot service
8. inspect logs
9. confirm bot responds

## Rollback flow

If migration fails before data mutation:

1. stop service if needed
2. inspect traceback
3. rollback code if needed
4. restore database backup if schema changed
5. rerun smoke

If migration succeeds but app fails:

1. inspect traceback
2. determine whether issue is code or schema
3. avoid speculative patching
4. rollback to previous commit and DB backup if needed

## Migration failure rules

If the same migration error appears twice:

- stop patching
- inspect migration file
- inspect generated SQL if needed
- inspect current DB schema
- decide rollback or isolated fix

Do not keep editing blindly.

## Current schema verification

Useful commands:

alembic current

alembic history

alembic heads

alembic upgrade head

alembic downgrade -1

For SQLite inspection:

sqlite3 data/cargopt.db ".tables"

sqlite3 data/cargopt.db ".schema transport_request"

For PostgreSQL inspection:

psql "$DATABASE_URL" -c "\dt"

psql "$DATABASE_URL" -c "\d transport_request"

## Smoke after initial migration

After initial migration, verify:

- all tables exist
- all core indexes exist
- all unique constraints exist where supported
- foreign keys exist
- minimal insert works
- event log insert works
- offer and assignment inserts work

Minimum smoke objects:

- carrier_company
- carrier_vehicle
- service_area
- location
- transport_request
- request_offer
- request_assignment
- request_event_log

## Branching policy

Migrations should be linear on main during MVP.

Avoid multiple Alembic heads.

If multiple heads appear:

- stop
- inspect alembic heads
- create merge migration only if necessary
- do not guess

## Environment policy

Local:

- may use SQLite
- may recreate database freely

Staging:

- must use separate database
- must test migrations before production
- must preserve realistic data when possible

Production:

- never reset database
- always backup before migration
- always run smoke after migration

## Initial MVP recommendation

For first implementation, prefer PostgreSQL-ready schema even if local SQLite is used.

Do not postpone core constraints too far.

Recommended initial migration scope:

- all core MVP tables
- all core foreign keys
- all core indexes
- all required unique constraints
- schedule-layer foundation tables

Do not include:

- payments
- ratings
- GPS tracking
- web admin-only tables
- external calendar integration tables

## Documentation alignment

Migration must match:

- 03_DB_Schema.md
- 14_SQLAlchemy_Models.md

If schema, models and migration disagree:

1. inspect exact conflict
2. update documentation if product decision changed
3. update model if implementation was wrong
4. regenerate or edit migration
5. rerun smoke

## Commit policy

Migration commits should include:

- ORM model changes
- migration file
- migration smoke if available

Do not commit migration without corresponding model changes.

Do not commit model changes that require schema update without migration.

## Final rule

A migration is not done when it is generated.

A migration is done only when:

- reviewed
- applied to empty DB
- smoke-tested
- downgrade considered
- committed with related model changes
